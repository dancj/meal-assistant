# Meal Assistant MVP

## Overview

Build the core pipeline for a meal planning assistant: recipes stored in Supabase, weekly meal plans generated by Google Gemini, delivered via Resend email. The MVP focuses on getting the end-to-end flow working — recipe storage, plan generation, and email delivery — before adding UI or automation.

All three services are free-tier. The app runs on Next.js 15 (App Router) deployed to Vercel.

## Problem Statement / Motivation

Household recipes are scattered across paper, bookmarks, AnyList, and Google Docs. Weekly meal planning is a manual chore. This tool automates it: store recipes once, get a weekly plan of 5 dinners + grocery list delivered to your inbox.

### Phase 3: Gemini Meal Plan Generation

**API route:** `src/app/api/generate-plan/route.ts`

**Flow:**
1. Validate `CRON_SECRET` (from `Authorization: Bearer <secret>` header)
2. Fetch all recipes from Supabase
3. Require minimum 5 recipes — return 400 error with count if fewer
4. Build prompt with recipe names, tags, and ingredients (exclude full instructions to save tokens)
5. Include dietary preferences (from env var `DIETARY_PREFERENCES` as JSON, or hardcoded for MVP)
6. Call Gemini with structured JSON `responseSchema`
7. Validate the response
8. Return the structured plan

**Gemini response schema:**

```typescript
// src/types/meal-plan.ts
export interface MealPlanDinner {
  day: string                    // "Monday", "Tuesday", etc.
  recipeName: string
  recipeId: string
  servings: number
  alternativeNote: string | null // e.g. "Kid alternative: plain pasta with butter"
}

export interface GroceryItem {
  item: string
  quantity: string
}

export interface MealPlan {
  dinners: MealPlanDinner[]      // exactly 5
  groceryList: GroceryItem[]
  weekOf: string                 // ISO date string
}
```

**Prompt strategy (pseudocode):**
```
You are a meal planning assistant for a household.
Select 5 dinners for the week from the following recipes.
Prioritize variety in cuisine, protein, and cooking method.

Recipes: [name, tags, ingredients for each recipe]

Dietary preferences: [household and per-person preferences]

For each dinner, note any alternatives needed for household members
with special dietary needs.

Generate a consolidated grocery list combining all ingredients,
deduplicating where possible.
```

**Serverless timeout:** Set `export const maxDuration = 60` in the route file. This requires Vercel's Fluid Compute (toggle in Vercel dashboard > Settings > Functions) which is available on the free tier and extends the timeout to 60s. This is sufficient for the Supabase fetch + Gemini inference + response parsing chain.

### Phase 4: Resend Email Delivery

**Utility:** `src/lib/email.ts` — function to format and send the meal plan email

**Flow:**
1. Accept a `MealPlan` object
2. Render it as HTML with inline styles (no Tailwind — email clients strip `<style>` tags)
3. Send via Resend to the comma-separated `EMAIL_RECIPIENTS` list

**Email content:**
- Subject: `Your Meal Plan for the week of [date]`
- Body: 5 dinner cards (recipe name, servings, alternative notes) + grocery list
- Simple, mobile-friendly inline-styled HTML

**Resend domain note:** The free tier sends from `onboarding@resend.dev` by default, which only delivers to the account owner's email. To send to other household members, verify a custom domain in Resend. Document this in the README setup instructions.

### Phase 5: Orchestration Endpoint

Wire phases 2–4 together in `src/app/api/generate-plan/route.ts`:

```
POST /api/generate-plan
  1. Validate CRON_SECRET
  2. Fetch recipes from Supabase
  3. Validate recipe count >= 5
  4. Generate meal plan via Gemini
  5. (Future: persist plan to Supabase meal_plans table)
  6. Send email via Resend
  7. Return { success: true, plan: { ... } }
```

Error handling:
- If Supabase is down → 500 with "Failed to fetch recipes"
- If Gemini fails → 500 with "Failed to generate plan"
- If Resend fails → 500 with "Plan generated but email failed" + include plan in response body so it's not lost
- All errors logged with enough detail for GitHub Actions to surface

## Technical Considerations

- **Vercel timeout:** `maxDuration = 60` with Fluid Compute enabled. Gemini inference typically takes 3–8s; total pipeline should complete in 15–20s.
- **Token budget:** For 50 recipes with name + tags + ingredients, expect ~5K–10K input tokens. Well within Gemini free tier limits (1M tokens/day).
- **Auth for MVP:** No user authentication. Recipe CRUD and generation endpoints are accessible by URL. Acceptable for a personal household tool deployed to an unlisted Vercel URL. The generation endpoint is protected by `CRON_SECRET`. Post-MVP, add Supabase Auth or a simple password gate.
- **Ingredient JSONB:** Using a JSONB column (not a separate table) for ingredients keeps queries simple and avoids joins. Good enough for a recipe collection under 1000 items.
- **Email HTML:** Must use inline styles. Keep it simple — no React Email or templating library for MVP. A plain template string with inline CSS is sufficient.

## Dependencies & Risks

| Risk | Mitigation |
|------|------------|
| Vercel 10s timeout | Enable Fluid Compute + `maxDuration = 60` |
| Gemini returns invalid JSON | Use `responseSchema` for structured output; validate response before emailing |
| Resend free tier domain restriction | Document custom domain setup; start with account owner email only |
| No auth on recipe endpoints | Acceptable for MVP (unlisted URL); track as known limitation |
| Gemini prompt produces poor plans | Iterate on prompt; start with a small recipe set to tune |
